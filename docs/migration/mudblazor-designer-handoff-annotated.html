Designer Handoff: MudBlazor Migration

This is the single static file to hand to the designer. The designer may edit this file directly or return annotated recommendations. Codex will later distribute the final decisions into AppTheme.cs, site.css, shared components, and migration rules.

Assignment For Designer
You are reviewing the LKvitai.MES Warehouse WebUI MudBlazor migration.

Goal:
Align the current MudBlazor bridge layer and future MudBlazor migration rules with the already approved LKvitai.MES design system. Do not invent a new visual language.

Primary design source:
- docs/ux/lkvitai-mes-ux-handoff.html
- portal-tokens.css
- portal-shell.css

Approved visual baseline:
- compact enterprise/product UI
- dark Portal topbar/header
- LKvitai.MES brand block
- neutral grey application surfaces
- teal active/focus/link/primary states
- warm sand only for operational warning/scaffold accents
- dense operational tables
- no marketing layout
- no decorative gradients
- no random bright colors
- no oversized cards

A. AppTheme.cs recommendations
Designer answer. The current palette is on-brand but under-mapped. MudBlazor reads ~20 palette fields when it auto-styles components; if they are left at Mud defaults you get stray purples/blues on links, text-secondary, dividers, action icons and the drawer. Map every field below to an existing token. Do not introduce any hex value that is not already in portal-tokens.css.

A3. Rules

  • Use --dark (#20242c) for both AppBar and Drawer so Mud surfaces match the Portal topbar exactly. The current #1f2632 (--n-800) is one step too light and reads as a different bar next to the Portal shell.
  • Info must be teal, not blue. Mud's default Info is #2196f3; mapping it to --info-fg keeps blue out of the product entirely.
  • Warning is sand (#7c5f2a), reserved for operational warn/scaffold only — never use Color.Warning for primary actions.
  • Set Button.TextTransform = "none" globally — the design system uses Title Case buttons; only chips/eyebrows/table headers are mono UPPERCASE.

C. MudBlazor component usage rules
Designer answer. MudBlazor is allowed as the engine, but Mud defaults are never the final look. Wrap operational surfaces in .lk / .lk-grid and skin them. Prefer semantic HTML for display-only content.

C2. Density & spacing

  • Always pass Dense="true" on grids, tables, app bars, lists.
  • Table row height 30px, cell padding 6×12. No density toggle.
  • Panel head padding 12×14; body 14. Toolbar padding 10×12.
  • Use the 4px spacing grid (4/8/12/16/20/24/28/32). Mud's Spacing param maps to 4px units — Spacing="3"=12px is the default gap.

C3. Hard rules

  • Never ship a page using Color.Primary/Secondary on chips or alerts — they bypass the token grammar.
  • Money, dates, dimensions, fabric codes, order ids → .mono + tabular-nums, always.
  • Icons at 18px inside Mud components; 16px inline in text. Neutral-500 at rest, teal-700 active.
  • No Elevation > 0 on operational surfaces. Dialogs/menus may use Mud's overlay shadow only.
D. Bootstrap migration rules
Designer answer. Replace Bootstrap surface-by-surface, leaf components first. Keep Bootstrap's grid (.row/.col) longest — it's layout-only and low-risk; remove Bootstrap's visual components (cards/alerts/badges/buttons) early because they carry the off-brand blue/shadows.

D2. Icons

  • Replace Bootstrap Icons with Mud's built-in Material icons (@Icons.Material.Outlined.*) at 18px, neutral-500 at rest. Outlined, not Filled, to match the line weight of the design system's CSS-drawn glyphs.
  • Keep the design system's mono-glyph data flags (€ ★ ! ↗ ↳) as-is — they are typographic, not icons, and must not be swapped for Material icons.
  • Do not load a third icon font. One Material set + the mono glyphs is the whole vocabulary.

D3. Coexistence (Phase 1–2)

  • Keep Bootstrap CDN + Bootstrap Icons loaded until Phase 3 (per Phase 0 rule), but scope all Mud overrides under .lk/.lk-grid so the two systems don't collide on bare elements.
  • Resource order matters: portal-tokens → portal-shell → Bootstrap → MudBlazor → site.css. site.css loads last so its !important bridges win over both Bootstrap and Mud. Keep this order.
  • Migrate a whole page at a time; don't leave a single page half-Bootstrap/half-Mud — mixed borders/radii read as broken.
E. Current conflicts
Designer answer. Conflicts found in the current Phase 0 implementation, ranked by risk before Bootstrap is removed in Phase 3.

E1. Token / palette conflicts

  • AppBar/Drawer use #1f2632 (--n-800), but the Portal topbar is #20242c (--dark). Side by side they look like two different bars. Fix in AppTheme (section A1).
  • Mud Info/Secondary still resolve to blue/pink wherever a component isn't explicitly colored (links inside MudText, selected nav, focus on some inputs). Mapping all palette fields (A1) closes this.
  • Bootstrap's --bs-primary (#0d6efd blue) and .btn-primary shadow are loaded globally and will bleed onto any un-migrated button. Acceptable in Phase 1–2 only because those pages are scheduled for replacement; flag any new page that uses a raw .btn.

E2. CSS specificity / structure

  • Current .lk-grid row height is 34px; design-system operational density is 30px. Standardise to 30 (section B7) so Mud grids match the static previews.
  • The bridge relies on !important against both Bootstrap and Mud. That's fine as a bridge, but it means un-scoped Mud overrides would also hit Bootstrap tables. Keep every override under .lk/.lk-grid/.panel — do not add bare .mud-* global rules.
  • Mud injects its stylesheet after Bootstrap but the bridge (site.css) loads last and wins — correct. Do not move site.css earlier or the bridge silently breaks.

E3. Component conflicts

  • SharedErrorAlert uses MudPaper + manual danger colors — good, but it should adopt the shared .lk-state--error grammar (B3) so error/warn/info/success are one system instead of a one-off.
  • SharedConfirmDialog hard-codes Color.Primary on the confirm button — fine once Primary = teal (A1), but the Cancel button uses Color.Default which renders Mud grey, not the --n-700 outline button. Add .lk wrapper + outlined skin (B4).
  • Material icon default size in Mud is 24px — too large for this density. Force 18px (C3).

E4. Phase 3 removal risks

  • When Bootstrap is removed, anything still using .row/.col, .d-flex, .text-muted, .mb-* utilities will lose layout. Audit for Bootstrap utility classes before dropping the CDN; replace with CSS grid/flex + token spacing.
  • Bootstrap Icons (.bi-*) glyphs vanish when the icon font is removed — every one must be migrated to Material (D2) first. Grep for bi- as a release gate.
  • Removing Bootstrap also removes its CSS reset; verify base box-sizing, heading margins and list resets exist in portal-tokens.css/site.css so type doesn't shift.

Answers To The Six Questions

  1. Palette complete? No — extend it (A1). The current 12 fields leave ~8 Mud fields at default (Info, Secondary, TextSecondary, Divider, Lines, Action, Table*), which is where stray blue/grey leaks in.
  2. Global vs local button skin? Skin globally under a .lk wrapper applied at the layout root, so every Mud button inherits the teal/dashed grammar without per-control classes. Reserve per-page classes only for exceptions.
  3. Final grid spec? 30px rows, 6×12 cells, mono-caps neutral header (--n-50 bg, --n-200 bottom border), hover --n-25, selected --accent-50 + 3px teal inset, pagination teal-600 active on --n-25 footer. Compact only, no toggle.
  4. Bootstrap replacements? card→MudPaper.panel, alert→.lk-state, badge→.chip, btn→MudButton, form-control→MudTextField/Select, table→MudDataGrid.lk-grid or .lk-table (see D1).
  5. Icons? Material Outlined via Mud at 18px (16px inline), neutral-500/teal-700; keep the mono data-glyphs. Drop Bootstrap Icons in Phase 3 (D2).
  6. Bridge CSS to change before Phase 1? Row height 34→30px; adopt .lk-state for SharedErrorAlert; set AppBar/Drawer to --dark; add Button TextTransform none + outlined Cancel skin.
